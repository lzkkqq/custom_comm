# custom_comm 目录设计

## 1. 概述

本文定义 custom_comm 仓库的目录结构的目标形态。文中"目标形态"指规划好、值得收敛到的结构；"现状"指 2026-04 的仓库快照。现状在第 3 节，目标在第 4 节，剩下章节解释差异、演进路径与外部对照。

三份相邻文档的职责：

- 本文 (dir-design.md)：定义"文件应该放在哪里，为什么"。
- refactor-plan.md：规定"从现状迁到目标形态"的阶段与 PR 边界。
- rebuild-plan.md：规定"新增一个算子的机械操作"，即脚手架 `tools/cc-newop` 的设计。

三者独立维护，若出现冲突以本文为准。读者按自己的关切选择入口：了解 why 与 what 读本文，了解 when 与 how 读 refactor-plan，要加新算子读 rebuild-plan。

## 2. 背景与约束

本仓的目录结构由三条不可协商的现实约束确定。这些约束在后面所有目录决策中反复出现。

### 2.1 ABI 边界

torch_npu 基于 `_GLIBCXX_USE_CXX11_ABI=1` 编译；CANN 9.x 的 libhcom/libhccl 基于 `_GLIBCXX_USE_CXX11_ABI=0` 编译。std::string、std::vector 这类常用容器在两种 ABI 下内存布局不同，跨 .so 直接传递会损坏内存。因此本仓无法只输出单一 .so，必须产出两个：

- `libcustom_comm_impl.so`（ABI=0）：链 libhcom/libhccl，提供 C 风格导出接口。
- `_C.cpython-*.so`（ABI=1）：链 libtorch_npu，走 torch_ext 注册。

两个 .so 之间只允许 POD 与 `extern "C"` 函数通信。这决定了目录里必须有一个"公共契约头"目录（本文称为 `include/custom_comm/`），作为 ABI 桥梁。

### 2.2 与 HCCL/CCU 的硬耦合

custom_comm 的所有 backend 最终都要调 HCOM/HCCL 接口，部分 backend（CCU_MS）还要链接 CANN 内部的 CCU kernel runtime。这部分代码天然与硬件绑定，无法脱离 Ascend NPU 单元测试。目录设计要把"硬件相关代码"与"纯 host 逻辑代码"分开，以便：

- 跑 ABI firewall / meta kernel / Python 层测试时，不强制启动 NPU 环境
- CCU kernel 与 HCCL 调用的变化只影响 backend 子目录，不外溢

这决定了 `ops/<op>/backends/<bname>/` 子目录必须是自足的单元。

### 2.3 与 omni-ops 的目标耦合

omni-ops 承诺作为 custom_comm 算子的发布载体。custom_comm 的 torch 层必须：

- 使用 omni-ops 接受的构建方式（setup.py + NpuExtension）
- 允许 omni-ops 用 `pip install custom_comm` 或 git submodule 方式引入，无侵入性依赖
- `torch.ops.custom_comm.<op>` 与 omni-ops 的其他 op namespace 共存

目录的顶层结构要便于 omni-ops 以不改内部的方式引入。

## 3. 设计原则

围绕约束确立的原则，有五条：

- SOLID-O（开放封闭）：加一个新算子或新 backend，只应新增文件，不应修改已有文件。这要求 TORCH_LIBRARY_FRAGMENT、静态注册机制、pkgutil 自动发现三套基础设施到位。
- SSOT：ABI 版本号、C API 声明、dtype 映射表、logging 宏等跨模块复用的定义，各自只有一个来源。见第 4.2 节的 `ops/_common/` 设计。
- 可组合构建：两个 .so 独立编译、独立打包，可以分别替换。下游消费者（比如 omni-ops）可以选择只用 Python API，也可以直接链 shim .so。
- 低意外成本：目录命名遵循"名字即职责"。`ops/` 是 shim 实现，`torch_ext/` 是 torch 绑定，`python/` 是 Python 包，`include/` 是 C 头。不设混合目录。
- 测试可分层：单元测试、集成测试、冒烟测试在目录上显式分开，CI 可按目录定义 matrix。

两个设计要点需要特别说明。

第一，`include/` 目录既放公共 C ABI 头（对 shim 与 extension 都可见），也是外部下游消费者（比如 omni-ops 的 C++ 组件）唯一可 include 的位置。这一层的稳定性要求最高：每次 breaking change 要 bump ABI 版本。

第二点，`ops/_common/` 与 `ops/_ccu/` 的下划线前缀表示它们是跨算子工具目录，不是算子本身。算子扫描工具（如 `tools/cc-newop`、CI 中的 op 注册检查）按 "不以下划线开头的一级子目录" 识别算子。

## 5. 分层职责

### 5.1 include/custom_comm/

职责：托管跨 ABI 契约的头文件。每个 C 头必须满足以下约束：

- 仅使用 C 语法（`extern "C" { ... }` 包围所有声明）
- 所有 struct 都是 POD，禁止 std::string / vector 等 C++ 类型
- 枚举有显式 tag 和整型大小（`typedef enum : int32_t {...} xxx_t;`）
- 版本宏统一在 `version.h`，格式 `CUSTOM_COMM_ABI_VERSION_{MAJOR,MINOR}`

进入这个目录的修改视同 ABI 变更，需要在 ADR 中讨论并更新版本号。

### 5.2 ops/（ABI=0 shim）

`ops/_common/` 与 `ops/_ccu/` 分别放不涉及具体算子的工具与 CCU kernel 基类。它们的分工：

- `_common/` 对所有 backend 都通用：registry、dtype 映射、环境变量读取、错误码转换、日志
- `_ccu/` 只对 CCU 族的 backend 有用：kernel 基类、GroupBroadcastItem、parallelDim 解析

每个算子的目录结构为：

```
ops/<op>/
├── inc/
│   └── internal.h            # 算子内部 C++ 头
├── api.cc                    # 实现 include/custom_comm/ops/<op>.h
├── dispatcher.{h,cc}         # 从 registry 选 backend
├── config.{h,cc}             # 运行时配置（例如 ENV var 解析）
└── backends/
    ├── decomposed/
    │   ├── strategy.cc
    │   └── internal.h
    ├── ccu_sched/
    │   ├── engine_ctx.{h,cc}
    │   └── kernel_mesh1d.{h,cc}
    └── ccu_ms/
        ├── engine_ctx.{h,cc}
        ├── kernel_v1.{h,cc}
        ├── kernel_v2.{h,cc}
        └── go_size.{h,cc}
```

加一个新 backend，就在 `backends/` 下加一个子目录，里头的 `strategy.cc` 在全局构造期调 `CC_REGISTER_BACKEND(...)` 把自己注册进 registry，dispatcher 通过 registry 查找、不需要改 `dispatcher.cc`。

### 5.3 torch_ext/ 层

```
torch_ext/
├── csrc/
│   ├── library.cc                    # 仅 TORCH_LIBRARY(custom_comm, m) 一个 stub
│   ├── common/
│   │   ├── tensor_bridge.{h,cc}    # at::Tensor ↔ cc_tensor_t
│   │   ├── comm_resolver.{h,cc}    # ProcessGroup → HcclComm
│   │   ├── stream_bridge.{h,cc}    # c10::Stream → aclrtStream
│   │   └── dtype_bridge.{h,cc}
│   └── ops/
│       └── allgather_batch.cpp    # TORCH_LIBRARY_FRAGMENT + eager + meta + (可选) converter stub
└── include/
    └── custom_comm_torch/           # 如需跨 op 共享 C++ 头，放这里
```

每个算子对应一个 `.cpp`。里面做三件事：`TORCH_LIBRARY_FRAGMENT(custom_comm, m)` 注册 schema；对 PrivateUse1 后端实现 eager kernel；对 Meta 后端实现 shape inference。

这里 schema 选择分散式注册（FRAGMENT），原因在 §7 有讨论。

### 5.4 `python/custom_comm/` 层

Python 侧最小化 wrapper，主要职责：

- 触发 libcustom_comm_impl.so 预加载（通过 `_loader.py` 的 `ctypes.CDLL`）。
- 基于 pkgutil 的自动发现，将 `ops/*.py` 内的函数挂到 `custom_comm.*` 命名空间。
- converters 下的 torchair 转换在 `import custom_comm.converters` 时才注册。

每个 `ops/<op>.py` 的内容约为：

1. 导入 `torch.ops.custom_comm.<op>`。
2. 定义 Python 签名的用户侧函数，处理 optional 参数、默认值、入参 dtype 检查。
3. 调用底层算子并返回结果。

converters 下的 `<op>.py` 定义 torchair converter，作用是把 `torch.ops.custom_comm.<op>` 节点翻译为 GE 图。这部分取决于算子在 graph 模式下的调用方式，可能需要多个 converter（不同 GE 算子）。

### 5.5 tests/

测试按执行环境分层，非按算子：

- `meta/`：torch meta device 上的 shape 推导。
- `unit/`：Python 层 / 参数校验等纯 CPU 测试。
- `eager/`：NPU 设备上的 eager 调用（单卡）。
- `graph/`：torchair 图捕获路径。
- `multi_rank/`：需要 2+ 卡的通信正确性测试。
- `size_boundary/`：多档位 size 覆盖测试（如 32K/48K/64K）。
- `bench/`：性能回归。
- `abi_firewall/`：用 nm / objdump 检查 .so 中符号归属。

每个目录下按算子分子目录。这种二维结构允许 CI 只跑 meta/unit（ABI 侧） 快速反馈，而把 eager/graph/boundary 放到夜间任务。

### 5.6 docs/

`docs/design/` 存设计文档（包括本文）。`docs/ops/<op>.md` 存算子的使用说明和 API 文档。`docs/decisions/` 存 ADR（architecture decision records），按日期 + 主题命名，例如 `2026-04-23-dual-abi-split.md`。

## 6. 与 omni-ops 的对照

本节按维度展示两个仓库的异同。

### 6.1 相同点

| 维度 | 具体做法 |
|------|----------|
| 算子 Python 包结构 | `<pkg>.ops.<op>`，pkgutil 自动发现 |
| torch schema | `TORCH_LIBRARY_FRAGMENT(<ns>, m)` 每算子一条 |
| build 系统 | setuptools + torch_npu NpuExtension |
| 测试栈 | pytest + torchrun |
| 命名风格 | snake_case 模块，CamelCase 类 |

### 6.2 差异点

| 维度 | custom_comm | omni-ops | 原因 |
|------|-------------|----------|------|
| 输出物 | 双 .so (shim ABI=0 + ext ABI=1) | 单 .so | HCCL 绑定的 ABI 差异 |
| C ABI 层 | 显式 `include/custom_comm/` | 无 | 跨 .so 边界需要 POD 契约 |
| kernel 路径 | CCU kernel + HCCL API | AscendC + ACLNN | 通信算子走 CCU，计算算子走 AscendC |
| backend 多态 | 每算子 `backends/<b>/` 目录 | 单一实现 | 通信场景需要多后端并存 |
| 集中注册 | 分散 FRAGMENT | 分散 FRAGMENT | （对齐） |
| Python converter | `converters/<op>.py` | `converters/<op>.py` | （对齐） |
| 测试分层 | 按执行环境 | 按算子 | 通信测试对硬件/多卡依赖重 |

omni-ops 侧当前是每算子独立目录并附带 csrc/converter 子目录。custom_comm 采纳相同的粒度但不同的目录骨架：算子下再分 backend，把跨 backend 共享的 helper 收进 `backends/` 内。

### 6.3 合并路径

如果 custom_comm 最终并入 omni-ops，目录兼容方式有三种：

方案 A：独立 pip 包 + 依赖。omni-ops 在其 requirements 声明 `custom_comm >= x.y`，两仓独立演进，API 通过 PyPI 对齐。custom_comm 内部结构不变。
方案 B：submodule vendoring。omni-ops 在内部 vendoring custom_comm（放 `third_party/custom_comm/`），共享构建。需要解决 submodule 的 ABI 链接问题。
方案 C：合并到 omni-ops 作为子目录。放 `omni-ops/<some-path>/custom_comm/`，CI 合并，release 合并。影响面最大。

当前建议走方案 A，直到 omni-ops 的构建框架明确支持双 .so 包为止。方案 C 作为长期目标。

## 7. 演进路径

这份目录是"北极星"，不是"明天就搬过去"的规划。目标形态分四步达成：

1. Phase 1（current focus，覆盖现有的 refactor-plan）：把 `ops/op_common/` 拆成 `ops/_ccu/` + `ops/_common/`；引入 `include/custom_comm/` 目录但只放公开头；所有 backend 注册走 registry；dispatcher 独立成文件。产物：可以加第二个算子。
2. Phase 2：`torch_ext/csrc/ops/<op>.cc` 拆分；`python/custom_comm/ops/` 包化；converters 拆分到独立模块。产物：加新算子 = 加新文件，不改旧文件。
3. Phase 3：`tests/` 按阶梯分层；ABI firewall 测试融入 CI；文档体系 (`docs/ops/<op>.md`) 建立。产物：CI 覆盖矩阵完整。
4. Phase 4：`tools/cc-newop` 脚手架上线（详见 rebuild-plan.md）。

每个阶段都有独立验收标准；refactor-plan 给出 PR 粒度的拆分。

## 8. 折中记录

为什么 TORCH_LIBRARY_FRAGMENT 分散而不集中？。集中注册意味着每加一个算子都要改中心 cpp（schema 表），违反 OCP。分散注册让每个算子的 cpp 自包含。唯一代价是 schema 检查没有集中入口，需要靠 `__init_subclass__` 等机制或测试覆盖。

为什么选择 _common 与 _ccu 两个独立目录而不合并？。_common 是所有 backend 都用的（dtype、logging、registry），_ccu 只有 CCU backend 用（LoopGroup、CalGoSize）。decomposed backend 不应被迫 depend on CCU 基础设施。分开可以让 decomposed backend 的最小依赖集更清晰。

为什么 backends 在 ops/<op>/backends/ 下而不是 ops/_backends/<op>_<bname>/？。同一个算子的 backend 间共享的类型（desc、context 结构）需要私有头。放在 ops/<op>/ 内部可以自然共享，不需要 export。

为什么不独立 cmake 子项目？。setup.py + NpuExtension 已经是 torch 生态的习惯做法。引入 CMake 会让 omni-ops 下游集成更复杂（omni-ops 自己用 setup.py）。现有的顶层 CMakeLists.txt 仅作为开发期的本地 sanity check，不参与 wheel 构建。

## 9. 未决项

以下问题尚未在本文中定论，将通过 ADR 决策。

- 是否需要给 `libcustom_comm_impl.so` 加版本号（SOVERSION）。当前只发单版本；若未来 custom_comm 作为独立 SDK 给 non-PyTorch 用户，需要 SOVERSION 语义。
- 是否把 `include/custom_comm/` 下的头同时暴露给 downstream（装上 `pip install custom_comm` 后是否放到 `sysconfig.get_path('include')` 下）。
- `tools/cc-newop` 脚手架的最终形态：生成粘贴式模板，还是通过 jinja 模板化。
- 测试目录中 `tests/graph/` 与 `tests/eager/` 分层下的 boundary / regression 是否需要再进一步切分。

## 10. 文档演进规则

本文档是活文档。每次 phase 推进（refactor-plan.md 中每个 phase 的 closure）需要：

1. 把新目录形态与旧的差异记入本文"变更记录"（位于文档末尾）。
2. 对每一项重要偏移，新增一份 `docs/design/decisions/NNNN-<title>.md`。
3. 若与 omni-ops 的接口契约发生变化，需要至少同步更新 omni-ops README。
