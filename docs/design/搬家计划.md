# custom_comm 搬家计划

- 版本: v1.0
- 日期: 2025-04-24
- 范围: 将 `omni-ops/custom_comm/` 整仓合入 `omni-ops/training/ascendc/`，完成后删除 `custom_comm/` 目录

## 一、背景

`custom_comm` 是一个独立维护的 torch 扩展，提供 `allgather_batch` 一个 HCCL collective op。它采用双 so 架构：`libcustom_comm_impl.so`（ABI=0，与 CANN 运行时对接）和 `_C.so`（ABI=1，与 torch 对接），通过 extern "C" POD 接口桥接。

`training/ascendc/torch_ops_extension/omni_training_custom_ops` 是 omni-ops 的主 torch 扩展包，目前通过 aclnn 封装了若干 ops-transformer 算子（mhc、mome 等）。它当前只有单 .so 结构（ABI=1），不涉及直接对接 CANN HCCL 的 ABI=0 代码。

本次迁移目标是把 custom_comm 的两层架构融入 omni_training_custom_ops，使用户调用从 `torch.ops.custom_comm.allgather_batch` 切换到 `torch.ops.custom.npu_ai_infra_all_gather_batch`，最终下线 `custom_comm/` 目录。

## 二、核心约束

CANN 的 libhccl、libhcom、libascendcl 三个运行时库是 ABI=0 （`_GLIBCXX_USE_CXX11_ABI=0`）编译的。torch_npu wheel 是 ABI=1 编译。两者共存于同一进程时，任何 std::string / std::vector 穿越边界都可能 segfault。

因此必须保留 custom_comm 现有的隔离设计：

- ABI=0 侧：纯 C++ 实现，直连 HCCL，打包成独立 .so
- ABI=1 侧：torch 扩展，暴露 TORCH_LIBRARY + pybind
- 两侧接口：extern "C" + POD struct，绝对不传递 std 类型

这个约束直接决定了本次迁移不能简单地把 custom_comm 的 .cpp 贴到 omni_training_custom_ops 下面就算完事。必须新增一个独立编译目标产出新的 ABI=0 shim。

## 三、决策汇总

前期讨论确认的关键决策：

| 项 | 决策 |
|---|---|
| ABI=0 shim 粒度 | 所有 HCCL 原生算子共用一个 `libomni_hccl_impl.so`，不为每个算子单独产 .so |
| shim 库命名 | `libomni_hccl_impl.so` |
| shim 源码位置 | `training/ascendc/src/ops-hccl/`（与 `ops-transformer/` 平级，新建） |
| 公共代码（原 `op_common`）位置 | `src/ops-hccl/common/`（独立子目录，shim 内部共享） |
| aclGraph stream 切换逻辑 | 保留原 thread_local 实现，不改 |
| 测试目录 | 跟随 `ops-transformer` 的布局：单元测试就近放算子目录，系统测试放 `src/tests/st/` |
| Python 入口 | `import omni_training_custom_ops; torch.ops.custom.npu_ai_infra_all_gather_batch(...)` |
| eager pybind 快路径 | 保留，暴露为 `omni_training_custom_ops.custom_ops_lib.all_gather_batch_inplace` |
| 算子命名 | torch op: `npu_ai_infra_all_gather_batch`；C++ 内部函数保留可读命名 |
| 是否删除 custom_comm | 最后一阶段删除；不做 deprecation shim |

## 四、目标架构

### 4.1 分层职责

```
用户代码
  │
  ├── torch.ops.custom.npu_ai_infra_all_gather_batch(...)    # graph path
  │       ↓ torch dispatcher
  │       ↓ TORCH_LIBRARY_IMPL(custom, PrivateUse1)
  │       ↓
  └── custom_ops_lib.all_gather_batch_inplace(...)            # eager fast path
          ↓ pybind11 binding
          ↓
     ─────────────────────────────────────────────────────── ABI=1 / ABI=0 边界
          ↓
     libomni_hccl_impl.so (ABI=0)
          ↓ extern "C" API + POD 结构
          ↓
     libhccl / libhcom / libascendcl (CANN 运行时)
```

两条路径共享同一个 C++ 实现函数。pybind 路径允许用户预分配 outputs 以避免 alloc 开销。

## 五、迁移阶段

分 5 个 PR 推进，每个 PR 可独立验证和回滚。

### Phase 1：Shim 独立构建

目标：把 `custom_comm/ops/` 的 C++ 代码搬到 `training/ascendc/src/ops-hccl/`，能独立编译出 `libomni_hccl_impl.so`，验证 ABI=0 契约。

动作：
1. 新建 `training/ascendc/src/ops-hccl/` 目录结构
2. 复制 `custom_comm/ops/op_common/*` → `src/ops-hccl/common/`
3. 复制 `custom_comm/ops/allgather_batch/*` → `src/ops-hccl/all_gather_batch/`
4. 编写 `src/ops-hccl/CMakeLists.txt`：聚合 `common/` 和各算子子目录，最终产出 `libomni_hccl_impl.so`
5. 在 `training/ascendc/CMakeLists.txt` 中 `add_subdirectory(src/ops-hccl)`
6. 编写一个独立的 C++ smoke test（不依赖 torch），验证 shim 的 extern "C" 接口能被调用

验证：
- `nm -D libomni_hccl_impl.so | grep __cxx11` 无输出
- `ldd libomni_hccl_impl.so` 显示只依赖 CANN 的 libhccl / libhcom / libascendcl
- smoke test 在 2 卡 NPU 机器上跑通

### Phase 2：ABI=1 wrapper + schema 注册

- 新建 `omni_training_custom_ops/ops_hccl/all_gather_batch/csrc/all_gather_batch.cpp`
- 写 TORCH_LIBRARY_IMPL (PrivateUse1, Meta)，实现同 custom_comm 的 torch_ext/csrc/allgather_batch.cpp
- 在 `csrc_base/ops_def_registration.cpp` 里增加 schema `m.def("npu_ai_infra_all_gather_batch(...) -> Tensor[]");`
- 修改 `torch_ops_extension/setup.py`：
    - `library_dirs` 增加 shim 的构建目录
    - `libraries` 增加 `omni_hccl_impl`
    - `extra_link_args` 加 `-Wl,-rpath,$ORIGIN`
    - `package_data` 把 libomni_hccl_impl.so 装进 wheel
- 修改 `omni_training_custom_ops/__init__.py`：
    - 在 `import` 之前 `ctypes.CDLL(os.path.join(os.path.dirname(__file__), 'libomni_hccl_impl.so'), mode=ctypes.RTLD_GLOBAL)`
    - 捕获异常，给出清晰错误（CANN runtime 缺失等）

验收：`import omni_training_custom_ops; torch.ops.custom.npu_ai_infra_all_gather_batch(...)` 可调用，在 aclgraph 捕获下也不 crash。

### Phase 3：pybind 快路径 + torchair converter

- 在 `csrc_base/ops_def_registration.cpp` 的 PYBIND11_MODULE 块里新增 `m.def("all_gather_batch_inplace", ...)`
- 实现函数：接受预分配好的 output tensor 列表，直接调 shim 的 extern "C"，不走 dispatcher
- 迁移 `custom_comm/python/custom_comm/converters/allgather_batch_converter.py` 到 `ops_hccl/all_gather_batch/converter/all_gather_batch.py`
- 对照 `ops_transformer/mhc/ai_infra_mhc_post_grad/converter/` 的写法调整 import 和注册方式

验收：eager path 延迟相比原 custom_comm 不劣化；torchair 图捕获与 custom_comm 原行为一致

### Phase 4: tests 迁移

- `custom_comm/tests/allgather_batch/*` → `training/ascendc/src/tests/st/ops_hccl/all_gather_batch/`
- conftest 合并到 `training/ascendc/src/tests/st/conftest.py`
- pytest marker / resources fixture 沿用 training/ascendc 既有约定（device / nodes / rank）
- 把 `custom_comm/tests/` 里和 CI pipeline 相关的 YAML / 脚本关联迁移

验收：`pytest training/ascendc/src/tests/st/ops_hccl/` 全绿。

### Phase 5：下线 custom_comm

- 删除 `omni-ops/custom_comm/` 整个目录
- 修改 `omni-ops` 顶层 CMakeLists、README、CI 配置，移除所有 `custom_comm` 引用
- 告知下游：`import custom_comm` 要改为 `from omni_training_custom_ops.ops_hccl import all_gather_batch`，`torch.ops.custom_comm.allgather_batch` 要改为 `torch.ops.custom.npu_ai_infra_all_gather_batch`

## 六、风险与应对

| 风险 | 等级 | 应对 |
|---|---|---|
| shim `.so` 与 torch 扩展 ABI 不兼容，符号解析失败 | 高 | CI 加 `nm -D \| grep __cxx11` 检查；rpath 设为 `$ORIGIN` 保证相对路径稳定 |
| torchair converter 签名变更未跟进 | 中 | Phase 3 先跑 `torch.compile` + aclgraph 两套端到端测试 |
| pybind 暴露的 fast path 与 dispatcher 分派行为不一致 | 中 | 两条路径共用 C++ 实现函数，只在 Python 入口分叉 |
| 迁移期间 custom_comm 和新路径同时存在，用户两头依赖 | 低 | 迁移完成后立刻删目录，不保留过渡 |
| Ascend C 构建体系对 src/ops-hccl/ 类 non-kernel 代码支持度未知 | 中 | Phase 1 先验证：写个最小 CMakeLists，能用 intf_pub 以外的纯 CMake target 编出 .so；若不行，回退到 torch_ops_extension 下用 CppExtension 自建 cmake 调用 |

## 七、Open Questions

- 是否把 shim 产物输出路径直接写死到 `torch_ops_extension/omni_training_custom_ops/` 下，还是用 install 阶段把 .so 拷过去？建议后者，更清晰。
- torchair converter 在 graph 捕获时如何处理 string 类 group_name 参数？需在 Phase 3 确认。
- 是否需要保留一个空的 `custom_comm/` package 作为 deprecation shim 让 `import custom_comm` 仍能工作并打出 warning？当前决策是不保留，干净删除。
