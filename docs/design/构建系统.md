# custom_comm 构建与脚手架重构计划

## 目标

把"新增一个算子"的边际成本从"改 8 处 + 写 4-5 份样板"降到"一条 `cc-newop.sh <name>` + 填 3 处语义 TODO"。核心改动：

1. **构建去中心化**（Stage 1）：setup.py 自动发现 `ops/*/inc`、`ops/*/src`，不再维护硬编码清单
2. **Torch 注册分散**（Stage 2）：`TORCH_LIBRARY_FRAGMENT` 让每个算子自管 schema，`ops_registration.cpp` 只留骨架
3. **Python 自动暴露**（Stage 3）：`ops/` 和 `converters/` 改为 pkgutil 动态 import
4. **脚手架 `cc-newop.sh`**（Stage 4）：从 `allgather_batch` 模板化拷贝 + 改名 + 注入 schema + 预埋 3 个 TODO

非目标：
- 不碰 ABI firewall（`libcustom_comm_impl.so` 的 C-only 边界）
- 不做 op registry 的跨进程动态加载
- 不生成 CCU kernel 的 DSL 代码（算法语义必须人工写）
- 不自动生成 torchair GE converter 的图 IR（仅给占位文件）

## 1. 当前基线（2026-01）

### 1.1 目录现状

```
ops/
├── op_common/
│   ├── inc/ccu_kernel_alg_base.h    CcuKernelAlgBase + BroadcastItem + CalGoSize(parallelDim)
│   └── src/ccu_kernel_alg_base.cc   GroupBroadcastBatch 基建
├── allgather_batch/
│   ├── inc/{allgather_batch,common,log_util}.h

│       ├── all_gather_batch.cc      # HCCL C API 入口
│       ├── ccu_dispatch.{h,cc}      # SCHED/MS 选择
│       ├── decomposed/              # path A: 分解 HCCL 实现
│       ├── ccu_sched/               # path B: SCHED CCU kernel
│       └── ccu_ms/                  # path C: MS CCU kernel

torch_ext/csrc/
├── ops_registration.cpp              # 集中式 TORCH_LIBRARY(custom_comm, m) { m.def(...); }

python/custom_comm/
```

### 1.2 新 op 今天要改哪些位置？

| 层 | 文件 | 手动强度 |
|----|------|---------|
| ABI 头 | `ops/<name>/inc/<name>.h` | 新建 |
| 实现 | `ops/<name>/src/...` | 新建（含 ccu_ms/ccu_sched/decomposed 三套） |
| setup 构建 | `setup.py` 的 `_SHIM_INCLUDES` 追加 | 硬编码改 |
| CMake | `CMakeLists.txt` 追加 `target_sources` 或 `file(GLOB)` | 硬编码改 |
| torch 注册 | `torch_ext/csrc/ops_registration.cpp` 的 `TORCH_LIBRARY` | 中心列表改 |
| torch 实现 | `torch_ext/csrc/<name>.cpp` | 新文件 |
| python wrapper | `python/custom_comm/ops.py` | 单文件 append |
| python `__init__` | `python/custom_comm/__init__.py` | 需导出新 symbol |
| torchair converter | `python/custom_comm/converters/__init__.py` 的 try import 链 | 中心列表改 |
| converter 实现 | `python/custom_comm/converters/<name>_converter.py` | 新文件 |
| 测试 | `tests/<name>/` + 至少 eager/size_boundary 两份 | 新目录 |
| ABI 防火墙测试 | `tests/test_abi_firewall.py` 若引入新 symbol 需校准 | 条件改 |
| bench（可选） | `tests/<name>/bench.py` + `ccu_ms_bench.py` | 新文件 |

一共 13 处。把这 13 处收敛到 "生成一次 + 人改 3 个 TODO"，就是本计划的核心。

### 1.3 最近提交揭示的硬约束

只看 commits `67af0a0`、`ccec016`、`15ed675`、`d3d9708`、`f9de692` 就能看到 scaffold 必须内嵌的"雷区"：

**I-1. `parallelDim` 三处必须一致**（MS kernel 的 GroupBroadcastBatch 分层切片）：

```
AllocGoResource(parallelDim) ⇔ CalGoSize(size, parallelDim) ⇔ GroupBroadcastBatch(..., parallelDim)
```

`d3d9708` 加了 parallelDim 到 CalGoSize 的形参；`67af0a0` 修正了 loopParam 位打包与 parallelDim 与硬件不一致。scaffold 生成的 MS kernel 模板必须把这个 constant 暴露在一处，三处引用。

**I-2. BroadcastItem 的 src/dst 资源独立**（`f9de692` 文档化）：同一个 BroadcastItem 数组里，所有 item 的 `src`、`dst` 必须来自调用方独立持有的资源。scaffold 模板里的 Algorithm() 骨架需要在注释里直接引用约束原文。

**I-3. size_boundary 必须覆盖 32KB 和 48KB**（`15ed675`）：LoopGroup[0] 的档位会在这两个尺寸暴露问题。scaffold 生成的 `test_size_boundary.py` 默认参数 `[32*1024, 48*1024, 96*1024]`，用户可以加不能删。

## 2. 四阶段改造

### 2.1 Stage 1：构建自愈（0.5 天）

- `setup.py::_SHIM_INCLUDES` 改为 `glob(ops/*/inc)` + `[op_common/inc]`
- `setup.py::_SHIM_SOURCES` 保持现有的 `rglob("*.cc")`，不改
- `CMakeLists.txt` 改为 `file(GLOB_RECURSE ops/**/*.cc)`，与 setup.py 同步

验收：在 `ops/dummy/` 下创建最小算子目录即编译通过，无需改 `setup.py`。

### 2.2 Stage 2：Torch schema 分散注册（0.5 天）

- `ops_registration.cpp` 改为空 library：`TORCH_LIBRARY(custom_comm, m) {}`
- 每个 op 的 `torch_ext/csrc/<op>.cpp` 里自己 `TORCH_LIBRARY_FRAGMENT(custom_comm, m) { m.def("…"); }`
- `TORCH_LIBRARY_IMPL` 同文件、同 library

tradeoffs：`TORCH_LIBRARY_FRAGMENT` 允许多处定义 schema，但同 schema 重复 `m.def` 会 crash。脚手架模板保证一个 op 只有一处 `m.def`。

### 2.3 Stage 3：Python 侧自动发现（0.5 天）

改动：
- 把 `python/custom_comm/ops.py` 拆成 `python/custom_comm/ops/<name>.py` 子包，每个文件定义 `__all__`。
- `python/custom_comm/__init__.py` 用 `pkgutil.iter_modules(ops.__path__)` 把所有 op 符号暴露到顶层。
- `python/custom_comm/converters/__init__.py` 同法自动 import 所有 `*_converter.py`（GE converter 通过装饰器注册，副作用即生效）。

验收：在 `python/custom_comm/ops/foo.py` 放一个 `def foo(...): ...`，`from custom_comm import foo` 直接成功。

## 3. cc-newop.sh 脚本详细设计

### 3.1 调用形态

```bash
./tools/cc-newop.sh reduce_scatter_batch \
    --schema "(Tensor[] inputs, int[] sizes, str hcom, int world_size) -> Tensor[]" \
    --backends ms,decomposed          # 默认 ms+decomposed；可选 sched
```

### 3.2 流程

1. **名字派生**：`reduce_scatter_batch` → `ReduceScatterBatch`（Pascal）/ `REDUCE_SCATTER_BATCH`（scream）/ `reduceScatterBatch`（camel）。
2. **从 `allgather_batch` 模板拷贝**：`cp -r ops/allgather_batch ops/<new>`、`cp -r tests/allgather_batch tests/<new>`、`cp torch_ext/csrc/allgather_batch.cpp torch_ext/csrc/<new>.cpp`、`cp python/custom_comm/ops.py → ops/<new>.py` 等。
3. **token 替换**（只替换有 word-boundary 的 identifier）：
    - `allgather_batch` → `<new>`（小写）
    - `AllGatherBatch` → `<Pascal>`
    - `ALLGATHER_BATCH` → `<SCREAM>`
    - `Allgather` → `<Pascal-word>`
4. **Schema 注入**：把 `--schema` 参数值写进 `torch_ext/csrc/<new>.cpp` 的 `m.def("..."`)。
5. **预埋 TODO**：在 `ops/<new>/src/ccu_ms/*_v2.cc` 的 `Algorithm()` 函数体里写注释 `// TODO(scaffold): 实现 CCU kernel 主体；参考 allgather_batch 的两段 LoopGroup 结构。注意 parallelDim 在 AllocGoResource/CalGoSize/GroupBroadcastBatch 三处必须一致（见 commits 67af0a0/d3d9708）。`
6. **列印 git status + 下一步指引**：告诉用户该改哪三处 TODO、pytest 怎么跑。

### 3.3 命名替换中的 hack-tricky

- **不要用 naive `sed`**：`allgather` 在 SDK API、注释、变量名里出现时应被替换；但 `Allgather` 在类名内部（如 `class AllGatherBatchOp`）也应替换。脚本必须先按词分词，再针对 `snake_case` / `PascalCase` / `UPPER_SNAKE` 三种形式分别替换。
- **跳过空间**：`op_common/`、`torch_ext/csrc/common/` 是通用基础，绝不替换其中的 symbol。
- **跳过 third-party 头**：`hccl/ccu_kernel.h`、`torch/library.h` 这类名字里可能碰巧含关键词，要白名单排除。
- **顺序敏感**：先替换长名（`allgather_batch` → `<new>`），再替换短名（`allgather` → `<new>_short_form`，如果 op 语义需要）；否则长名会被半替换。
- **大小写保持**：生成新名字时要根据 context 生成正确的 PascalCase（`all_gather_batch` → `AllGatherBatch`），常见错误是 `All_Gather_Batch`（下划线未吃掉）或 `Allgatherbatch`（首字母大写但词界丢失）。

### 3.4 三个不能自动生成的 TODO

scaffold 不碰这三处，让用户显式填写：

1. **desc 结构**（`ops/<op>/inc/<op>.h`）：算子的输入 / 输出描述符。每个算子的数据结构不同，模板只给 `struct HcclXxxBatchDesc { /* TODO */ };`。
2. **kernel body** (`ops/<op>/src/ccu_ms/ccu_kernel_<op>_mesh1d_ms.cc::Algorithm()`)：CCU 内核的 BroadcastItem 构造 + GroupBroadcastBatch 调用。模板给出骨架 + 常见 pitfalls 注释，但核心 `items.push_back(...)` 要作者写。
3. **torchair converter** (`converters/<op>_converter.py`)：IR 图模式下要把这个 op 展成什么 GE 算子。没有通用规则。

## 4. 不变式（scaffold 生成代码必须体现）

所有 scaffold 生成的代码必须体现以下 3 条不变式（对应最近几次 bug fix）：

### 4.1 I-1. parallelDim 单一来源
- `ccu_kernel_<op>_mesh1d_ms.cc` 里生成 `static constexpr uint32_t kParallelDim = 64;` 作为常量
- `Algorithm()` 里统一使用 `kParallelDim`，不出现字面量
- `AllocGoResource(...)` 和 `CalGoSize(...)` 都必须从 `kParallelDim` 读

对应 commit 67af0a0、d3d9708

### 4.2 I-2. BroadcastItem 资源独立
- 模板注释明确要求：同一次 `GroupBroadcastBatch` 里各个 item 的 src/dst handle 必须互不别名
- 如果 `items[i].src == items[j].src`，需要在上层先 fork 出独立 handle

对应 commit `f9de692` 的 docs 说明。

### 4.3 I-3. size_boundary 覆盖

`tests/<name>/test_size_boundary.py` 的 parametrize 必须包含 32KB、48KB（commit 15ed675 的教训）。scaffold 写死这两档，用户可以追加但不能删。

```python
@pytest.mark.parametrize("nbytes", [
    32 * 1024,  # LoopGroup[0] 下界
    48 * 1024,  # LoopGroup[0] 上界
    128 * 1024,
    1 << 20,
])
def test_<name>_size_boundary(dist_ctx, nbytes): ...
```

## 5. 验收

- Stage 1 完成：`ops/foo/inc/foo.h` + `ops/foo/src/foo.cc` 加进项目不改 setup.py，编译通过。
- Stage 2 完成：`torch_ext/csrc/foo.cpp` 里只用 `TORCH_LIBRARY_FRAGMENT`，不动 `ops_registration.cpp`，`torch.ops.custom_comm.foo` 可访问。
- Stage 3 完成：`python/custom_comm/ops/foo.py` 存在即可被 `from custom_comm import foo` 拿到。
- Stage 4 完成：`cc-newop.sh my_new_op` 一键生成，`pytest tests/my_new_op -k meta` 立即绿（只 meta，kernel 填 TODO 时当然失败）。

## 6. 风险与回退

- 若 `TORCH_LIBRARY_FRAGMENT` 在当前 torch 版本行为异常（旧版本曾有 fragment 重复注册 bug），回退到单一 `ops_registration.cpp` 集中 `m.def` 的方案，改用脚本在 scaffold 时自动 patch 这个文件。
- 若 `pkgutil.iter_modules` 在 `pip install -e .` 下找不到子模块（需要 __init__.py 空壳），回退为显式 import list，生成器负责维护。
- 若 scaffold 生成的 kernel 模板在"parallelDim=64 常量"之外需要动态调整，降级到在 TODO 处放 `// FIXME: override parallelDim if your op diverges`。

## 7. 不做的事

- 不生成"自适应 parallelDim"机制（不值得，kernel 语义各异）
- 不搞跨语言 schema 模板（ATen schema 和 C++ struct 的映射写手工就够）
- MS kernel 不再区分 V1/V2 — scaffold 只生成基于 `ccu_base::GroupBroadcastBatch` 的单一实现
- 不改 ops_registration.cpp 之外的 ABI 定义
